First run

Quick start — five steps

  1. Install software — Civil 3D + Dynamo (see Installation).
  2. Build the template drawing once — layers, styles, assembly, blocks, surfaces (First-time setup).
  3. Prepare data — copy civil3d_automation/csv/templates/*.csv into civil3d_automation/csv/, fill rows; copy config/project.example.json to config/project.json and edit names/paths. Plain-language field help: project.schema_notes.md.
  4. Optional preflight — run civil3d_automation/tools/run_preflight.bat to check CSV headers and copy the absolute path to project.json for Dynamo.
  5. Run in Civil 3D — open your project DWG, run Dynamo Player with m8_run_all.py and paste the JSON path; then check out/ (full run order).

New to road terms? Read road_automation_beginners.html.

Templates

Data starter kit and preflight helper

Blank tables — Header-only CSVs live in civil3d_automation/csv/templates/ (see templates/README.md). Copy them into csv/ so you keep the demo samples intact if you prefer.

Example config — civil3d_automation/config/project.example.json is a clean starting point for project.json.

Windows preflight — The small Python tool under civil3d_automation/tools/ checks that files referenced in project.json exist, validates CSV column headers against what M1–M7 expect, opens your csv or config folder, and copies the full path to project.json for Dynamo Player. Details: tools/README.md. If the window does not open, your Python may lack Tkinter — use python road_automation_preflight.py --validate path\to\project.json from a terminal instead.

Excel workflow — A multi-sheet starter workbook can be generated from the same headers; see templates/README.md (Excel section).

Overview

What this automation is

This workflow targets Autodesk Civil 3D projects where repetitive tasks (alignments from tables, profiles from grades, corridor rebuilds, volumes, signage placement, BOQ rollups, and pavement marking solids) are driven by CSV data and Dynamo Python Script nodes instead of hundreds of manual picks and dialog steps.

Two documentation layers work together:

  • Marking module — described in road_marking_dynamo.html: one Python script creates SN-style dashed/solid marking solids from a 3D polyline or feature line, with XData for re-run without duplicate geometry.
  • M1–M8 package — implemented under civil3d_automation/: separate scripts for alignment, profile, corridor regions, volumes, signage, sheet helper, BOQ rollup, and an orchestrator (m8_run_all.py) that reads config/project.json.
Important: Nothing here replaces engineering judgement. Automation applies your standards (CSV numbers, template styles, pay-item map) and still needs you to verify geometry, contract mapping, and issue control.
Software

Installation and prerequisites

There is no Windows “installer” executable in this repository. You install Autodesk products and then link this folder to your Dynamo workflow.

1. Required software

ComponentRequirementNotes
Civil 3D 2022 or newer (recommended) API names differ slightly by year; verify in civil3d_automation/API_NOTES.md after first run.
Dynamo Version bundled with Civil 3D (Dynamo 2.x) Use Civil’s Dynamo, not Dynamo Sandbox.
Python in Dynamo IronPython 2.7 for the supplied scripts In the Python Script node settings, set engine to IronPython 2 unless you port scripts to CPython 3 (different references and limits).

2. Repository / folder “installation”

  1. Copy or clone the project folder (e.g. d:\road desgin automation\) anywhere on disk.
  2. Keep civil3d_automation\ next to the HTML docs so relative links in this guide stay valid.
  3. Edit civil3d_automation\config\project.json and CSV files under civil3d_automation\csv\ for your job.
  4. When using M8, pass the absolute path to project.json into the Dynamo graph (e.g. D:\…\civil3d_automation\config\project.json).

3. Optional tools

  • Excel — for opening BOQ and volume CSVs; Power Query can reshape exports.
  • Git — to version CSVs and project.json per drawing issue.
No pip install inside Civil 3D for the default path: IronPython does not use openpyxl or yaml in these scripts. Configuration uses JSON (project.json). BOQ output is CSV for maximum compatibility.

One-time

First-time setup — template drawing

Automation assumes a template DWG or DWT that already contains Civil styles and CAD primitives the scripts reference by name. Full checklist is in civil3d_automation/template/README.md. Summarised here:

  • Layers: e.g. 0, C-ROAD-MARK, C-SGN-FURN (match project.json → names.layers).
  • Alignment & profile styles + label sets: names must match project.json → styles.* (defaults often Standard).
  • Assembly: an assembly whose name matches names.assembly (used by M3).
  • Blocks (M5): block definitions exactly as listed in signage_schedule.csv (e.g. SGN_STOP).
  • Surfaces (M4/M7): TIN (or compatible) surfaces for existing ground and finished model, named e.g. EG and FG per config.
  • Feature lines (optional M3 extension): if you later bind targets from CSV, name them consistently (e.g. EOP_L).

Creating this template is manual Civil 3D work once per firm or project type; afterwards each job mainly edits CSV/JSON and runs Dynamo.

Dynamo Player

Dynamo graphs — how to build them

Follow civil3d_automation/dynamo/README_BUILD_GRAPHS.md in detail. Short version:

Open Dynamo from Civil 3D
Manage tab → Dynamo (or Dynamo Player after graphs are saved).
New graph → Python Script node
Set engine to IronPython 2. Paste the entire contents of the matching python/m*.py file.
Wire inputs in order
Each script documents IN[0], IN[1], … at the top. Use String nodes for paths and names; Number sliders where needed.
Watch the output
Connect the Python output to a Watch node. Messages beginning with OK: or ERROR: tell you whether to commit or fix data.
Save and expose to Dynamo Player
Save as M1_Alignment.dyn, etc., under civil3d_automation/dynamo/ (optional but recommended for repeatability).
Operations

Steps to follow each design run

Option A — Full chain (M8)

  1. Open your project DWG created from the template.
  2. Update csv\ files and config\project.json (alignment PI, profile PVIs, regions, signage, pay items, output paths).
  3. Open Dynamo Player and run the graph that contains m8_run_all.py with IN[0] = full path to project.json.
  4. Optional IN[1]: comma list of steps, e.g. m1,m2,m4 to run a subset.
  5. For each marking source line, run the marking graph from road_marking_dynamo.html (between corridor work and signage/BOQ is typical).
  6. Open out\ CSVs in Excel; perform design QA and issue PDFs/sheets as per your QA procedure.

Option B — Step-by-step (M1 then M2 …)

  1. M1 — PI CSV → alignment (polyline-style fixed lines unless you extend curves).
  2. M2 — PVI CSV → layout profile on that alignment; read grade QA in Watch.
  3. M3 — Region table → corridor baseline regions + rebuild.
  4. M4 — Rebuild corridor; create volume surface EG vs FG; write volume CSV.
  5. Marking — per feature line / 3D polyline (hub HTML script).
  6. M5 — Signage CSV → blocks (re-run safe: prior XData-tagged inserts are erased).
  7. M6 — Checklist for sheets/publish (extend when your Civil API path is fixed).
  8. M7 — BOQ CSV from volumes + marking layer solids + signage layer counts.

Per-module detail

ModuleWhat the script automatesWhat stays manual
M1 Reads PI CSV; creates alignment; chained fixed lines; optional design speed stamp; duplicate-name check. PI locations and radii in CSV; true curve/spiral layout if not using line-only mode; template styles.
M2 Creates FG profile by layout; adds tangents from CSV; grade QA messages; attempts vertical curves when length > 0. Profile/layer/style names in template; engineering acceptance of grades and curves.
M3 Adds baseline regions from CSV; corridor rebuild. Assembly design; per-subassembly widths and offset targets (extend API).
M4 Corridor rebuild; TIN volume surface; cut/fill/net CSV. Surface definitions, EG/FG correctness, corridor model QA.
Marking Dash pattern from standard table; solids; layer; XData; delete-on-regenerate. Source line geometry and Z; type selection; optional width override.
M5 Erases prior run (same XData app); places blocks; tangent-based rotation. Block definitions; schedule content; offset/sign codes.
M6 Checklist string and optional output folder creation. View frames, sheet sets, publish — implement when API chosen.
M7 Merges volume CSV + marking solids on layer + signage counts into BOQ CSV via pay-item map. Pay-item dictionary; verifying quantities against contract; corridor material API extension if required.
M8 Loads JSON; resolves paths; runs M1–M7 in order; appends QA log. Correct absolute path to JSON; fixing any step that returns ERROR.
Reference

Quick module reference

  • M1 — python/m1_alignment_from_csv.py — Inputs: CSV path, alignment name, style, label set, start station.
  • M2 — m2_profile_from_csv.py — Inputs: CSV, alignment name, profile name, layer, profile style, label set, max/min grade for QA.
  • M3 — m3_corridor_regions_from_csv.py — Inputs: CSV, corridor name, default assembly name.
  • M4 — m4_corridor_rebuild_volumes.py — Inputs: corridor, EG surface name, FG surface name, output CSV path, optional volume surface name.
  • M5 — m5_signage_from_csv.py — Inputs: CSV, alignment, layer, optional XData app name.
  • M6 — m6_sheets_helper.py — Inputs: alignment name, output directory (informational).
  • M7 — m7_boq_rollup.py — Inputs: optional M4 volumes CSV path, marking layer, signage layer, payitems CSV, BOQ output CSV.
  • M8 — m8_run_all.py — Inputs: absolute project.json path; optional step filter string.

Exact IN[…] order is always at the top of each .py file.

Support

Troubleshooting and versioning

  • ERROR: style / layer / alignment not found — Open template; align names with project.json and the script inputs.
  • ERROR: alignment or profile already exists — Rename in Civil or delete the object before re-running M1/M2.
  • M4 volume fails — Confirm both surfaces are valid TINs (or types accepted by TinVolumeSurface.Create for your version).
  • M5 missing blocks — Insert block definitions into the template; names must match CSV exactly.
  • After Civil 3D upgrade — Re-test one module at a time; update API_NOTES.md with any signature changes.
Single source of truth for numbers: keep CSV and JSON under revision control with the drawing issue. The automation replay is only as good as the data you feed it.
